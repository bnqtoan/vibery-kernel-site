The request lifecycle

How a request flows

The declaration is the easy part to see — this is the depth underneath it. Every request travels a complete backend: identity, tenancy, authorization, validation, persistence, events, audit, and error handling. The only stage that is your code is the PolicyHook.

All three front-ends — REST, MCP, and CLI — converge on a single verb-execution path (resource.ExecVerb), so every caller travels the exact lifecycle below.

1
request → correlate
A request-ID is stamped onto the context so every log line, event, and audit row downstream is traceable to this call.
2
authn
The bearer token is resolved to a Principal — who is calling, with which capabilities.
3
scope
The Principal's tenant context is set on the context — the single source of truth every scoped read and write reads from.
4
store
An RLS-scoped connection is opened: Postgres runs SET app.current_tenant_id on the connection. Fail-closed — no tenant, no rows.
5
authz
The route maps to an IAM action; the policy is evaluated deny-wins. A request that isn't explicitly allowed is rejected.
6
validate
The request body runs the field validation pipeline before any business logic touches it.
7
PolicyHook
The only stage that is your application code. Your domain rule runs — e.g. checkRefundable — deciding whether this specific verb may proceed.
8
store
The write executes tenant-scoped, enforced by RLS at the database. The connection physically cannot reach another tenant's rows.
9
bus
On success, bus.Publish emits the verb's event to NATS JetStream — durable, for any consumer to react to.
10
journal
An append-only audit row records what happened, by whom, under which request-ID. Immutable by construction.
×
httperr
Any failure at any stage above short-circuits to the canonical error envelope — one consistent error shape across every resource.

Tenant isolation by default

Postgres Row-Level Security. Every connection stamps app.current_tenant_id, and a query with no tenant context fails rather than running unscoped. Tenant B cannot read tenant A's rows — enforced by the database, not by app filters. The BYPASSRLS pool is the only escape, and it must be explicit.

Durable events, not fire-and-forget

NATS JetStream. Every verb success publishes an event, and durable pull consumers ack/nak with redelivery — nothing is silently dropped. The audit journal is itself an event consumer, and automation (trigger) reacts to events without ever touching the producer.

MCP-native by construction

The dispatch package is an MCP Code Mode server with an embedded Goja JS sandbox. Every resource verb becomes an AI-callable tool automatically; the MCP contract is emitted from resource definitions, never hand-written.

Two first-class readings

Built for the architect and the strategist

Same kernel. One view shows the engineering guarantees; the other shows why those guarantees are leverage. Switch below.

RLS-by-default, fail-closed

A query with no tenant context fails — it never silently runs unscoped. Isolation is enforced at the Postgres layer, not in application if-statements.

PBAC, not role checks

Policy-based access control: policies are collections of allow/deny statements (AWS-IAM-style, with wildcard actions and resources), deny wins unconditionally, grants are scope-clamped and narrowed by constraints — evaluated per request, never role-name matching.

Headless by construction

The kernel ships zero UI — pure Go packages you mount on your own transport. The same verb lifecycle serves REST and MCP from one definition, so one backend serves your web app, your mobile app, a CLI, and AI agents identically. No framework lock-in on the frontend.

One error envelope, one event shape

Opinionated core: one way to scope a tenant, one httperr envelope, one event contract — so every resource is observable and debuggable the same way.

MCP-native by construction

Every verb registers an MCP tool through dispatch — Code Mode with a Goja JS sandbox. Your REST API is the agent API; there is no second surface to maintain.

Tested against real infra

266 test functions run against ephemeral Postgres + NATS spun up per test via testcontainers — not mocks. A 1:1 prod-to-test line ratio across ~7.2K lines each.

Escape hatches at the edges

The core is one-way, but you can always drop to raw SQL, a custom handler, or a custom MCP tool. Opinionated where it pays; open where you need it.

Mechanism, not policy

The kernel ships the machinery — tenancy, events, audit, the metering pipeline. Your app supplies the rules. Pricing and workflow steps never enter the kernel.
